# ADR-001: 选择管道式架构而非事件驱动架构

## 状态

已接受

## 背景

SynchroLens 是一个 AI 同声传译助手，核心数据流为：音频采集 → 语音识别 → 翻译 → 字幕渲染 → 笔记写入 → 纠正检测。系统需要满足以下约束：

- **低延迟**：字幕延迟 < 3s，STT 首字延迟 < 500ms，翻译首 token 延迟 < 1s
- **流式处理**：STT 逐词输出，LLM 逐 token 输出，中间结果需实时呈现
- **回溯纠正**：已输出的翻译可能需要事后修正
- **72 小时限时开发**：架构复杂度必须可控

在架构选型阶段，我们评估了三种方案：

| 方案 | 核心思路 | 优点 | 缺点 |
|------|----------|------|------|
| **管道式架构** | 各阶段顺序连接，数据单向流动 | 简单直观、延迟可预测、调试容易 | 阶段间耦合需显式定义接口 |
| **事件驱动架构** | 各模块通过事件总线发布/订阅 | 高度解耦、可动态增减模块 | 事件溯源复杂、调试困难、延迟不可预测 |
| **中心调度架构** | 中央调度器协调所有模块 | 全局视角、灵活调度 | 单点瓶颈、调度逻辑膨胀、开发量大 |

## 决策

采用**管道式架构**（Pipeline Architecture），数据流为：

```
音频采集 → STT引擎 → 原文流(逐词) → 翻译引擎 → 当前句翻译(实时可纠正)
→ 字幕渲染 → 句子确认(冻结) → 笔记写入(Markdown)
→ 纠正检测(每5句批量) → 笔记纠正脚注
```

## 原因

1. **延迟可预测且最优**：管道式架构中数据单向流动，每个阶段的处理时间可直接叠加，总延迟 = Σ(各阶段耗时)。事件驱动架构因事件队列和异步调度引入不可控延迟，无法满足 < 3s 的字幕延迟要求。

2. **流式处理天然适配**：STT 逐词输出和 LLM 逐 token 输出都是典型的流式数据，管道的"生产者-消费者"模型直接对应。每个管道阶段可以独立处理流式数据块，无需额外的事件序列化/反序列化开销。

3. **回溯纠正有明确边界**：管道中"句子确认（冻结）"是一个天然的分界点——冻结前的纠正走字幕实时更新，冻结后的纠正走笔记脚注。事件驱动架构中，纠正事件的时序和作用范围难以界定，容易导致状态不一致。

4. **72 小时开发周期下的最优解**：管道式架构代码量最少、心智模型最简单、调试路径最短。事件驱动架构需要实现事件总线、事件溯源、幂等处理等基础设施；中心调度架构需要编写复杂的调度逻辑。在限时比赛中，架构复杂度直接决定交付风险。

5. **模块解耦但不过度设计**：管道各阶段通过明确的接口（TypeScript 类型定义）连接，每个阶段可独立开发和测试。相比事件驱动架构的完全解耦，管道式架构在保持模块边界的同时避免了过度抽象。

## 后果

**正面：**
- 延迟可预测，便于性能优化和瓶颈定位
- 代码结构清晰，新成员可快速理解数据流
- 各阶段可独立单元测试，集成测试路径明确
- 72 小时内可完成核心管道的开发和联调

**负面：**
- 新增处理阶段需修改管道定义，不如事件驱动架构灵活
- 管道阶段间的数据格式需预先约定，变更时需同步修改上下游
- 若未来需要支持多路音频源或多语言并行翻译，管道式架构需要重构为多管道模式

**缓解措施：**
- 管道阶段间使用 TypeScript 接口定义数据格式，变更时编译器自动提示受影响的位置
- 预留管道配置能力，未来可通过配置文件动态组装管道阶段
