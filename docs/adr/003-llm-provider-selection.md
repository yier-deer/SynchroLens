# ADR-003: 选择 DeepSeek 作为翻译 LLM

## 状态

已接受

## 背景

SynchroLens 需要将 STT 输出的原文实时翻译为目标语言，核心需求为：

- **流式输出**：逐 token 输出翻译结果，实现"边翻边显"
- **低首 token 延迟**：翻译首 token 延迟需 < 1s（PRD 要求）
- **翻译质量**：同声传译场景要求翻译准确、流畅、符合语境
- **成本可控**：72 小时比赛期间需大量调用，成本不能成为瓶颈
- **API 简单易接入**：开发时间有限，接入复杂度直接影响交付

我们评估了三个候选方案：

| 方案 | 流式输出 | 首 token 延迟 | 翻译质量 | API 兼容性 | 价格（每百万 token） |
|------|----------|---------------|----------|------------|---------------------|
| **DeepSeek V4 Pro** | 支持 | ~400ms | 优秀 | OpenAI 兼容 | 输入 ¥2 / 输出 ¥8 |
| **智谱 GLM-5** | 支持 | ~500ms | 良好 | 自有 API | 输入 ¥5 / 输出 ¥50 |
| **通义千问 Qwen-Max** | 支持 | ~600ms | 良好 | OpenAI 兼容 | 输入 ¥20 / 输出 ¥60 |

## 决策

采用 **DeepSeek V4 Pro** 作为翻译 LLM。

## 原因

1. **性价比极高**：DeepSeek V4 Pro 的定价约为 GPT-5.5 的 1/12，在 72 小时高密度调用场景下，成本几乎可以忽略。相比之下，通义千问 Qwen-Max 的输出价格是 DeepSeek 的 7.5 倍，在大量流式调用下成本显著。

2. **首 token 延迟满足要求**：DeepSeek V4 Pro 首 token 延迟约 400ms，低于 PRD 要求的 1s，结合讯飞 STT 的 < 80ms 首帧延迟，总延迟远低于 3s 的字幕延迟上限。

3. **OpenAI 兼容 API 降低接入成本**：DeepSeek V4 Pro 的 API 完全兼容 OpenAI Chat Completions 格式，可直接使用成熟的 OpenAI SDK（或兼容客户端），无需学习新的 API 规范。智谱 GLM-5 使用自有 API 格式，需要额外的适配代码。

4. **流式输出稳定**：DeepSeek V4 Pro 的流式输出（SSE）实现稳定，token 粒度均匀，适合逐 token 渲染到字幕界面。在管道架构中，翻译引擎阶段可以直接将流式 token 推入字幕渲染阶段。

5. **翻译质量满足同传场景**：DeepSeek V4 Pro 在中英互译任务上的表现接近 GPT-4 级别，对于同声传译中常见的口语化表达、专业术语、中英混合输入均有良好的翻译能力。在限时比赛场景下，翻译质量的微小差异远不如延迟和成本重要。

6. **72 小时开发效率**：OpenAI 兼容 API 意味着可以使用现有的 TypeScript OpenAI SDK，接入代码量极少。流式处理只需监听 `stream` 事件，与管道架构的翻译引擎阶段无缝对接。

## 后果

**正面：**
- 翻译首 token 延迟 ~400ms，端到端延迟（STT + 翻译 + 渲染）远低于 3s 上限
- 成本极低，72 小时内可无限制调用，不影响比赛预算
- OpenAI 兼容 API 使接入代码简洁，开发效率高
- 流式输出与管道架构天然适配

**负面：**
- DeepSeek 服务偶尔出现高峰期排队，可能导致首 token 延迟波动
- 依赖单一服务商，若 DeepSeek 服务不可用则翻译功能完全中断
- DeepSeek 的翻译风格偏向书面语，口语化场景可能需要 prompt 工程优化

**缓解措施：**
- 在翻译 prompt 中明确要求"口语化翻译"，适配同声传译场景
- 实现超时重试机制：首 token 超过 2s 时自动重试
- 预留 LLM Provider 抽象层，未来可快速切换到其他 OpenAI 兼容服务商（如通义千问）
