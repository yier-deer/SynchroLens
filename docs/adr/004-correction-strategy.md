# ADR-004: 纠正策略——字幕只管当前句，笔记承载纠正

## 状态

已接受

## 背景

SynchroLens 的管道架构中，STT 和翻译都可能产生需要纠正的结果：

- **STT 纠正**：讯飞实时转写会先输出中间结果（如"你好世"），随后纠正为完整结果（"你好世界"）
- **翻译纠正**：LLM 翻译可能在上下文更完整后产生更优翻译
- **批量纠正**：CorrectionDetector 每 5 句触发一次批量检查，可能发现早期翻译的改进空间

关键问题是：**纠正结果如何呈现给用户？** 我们评估了三种呈现方式：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **字幕实时回溯** | 纠正时直接修改已显示的字幕内容 | 用户始终看到最新结果 | 视觉闪烁、阅读中断、已读内容突然变化造成困惑 |
| **字幕只管当前句 + 笔记脚注** | 未冻结句子实时纠正，已冻结句子的纠正写入笔记脚注 | 字幕稳定、笔记完整、体验流畅 | 纠正不在字幕中直观可见 |
| **纯追加模式** | 不做任何纠正，原样输出 | 最简单 | 不满足比赛"修正能力"要求，翻译质量无法提升 |

## 决策

采用**字幕只纠正当前句（未冻结），已冻结句子的纠正写入笔记脚注**的策略。

具体规则：
1. **未冻结句子**（当前正在翻译的句子）：字幕实时更新，纠正直接反映在字幕上
2. **冻结句子**（用户已确认或自动确认的句子）：字幕不再修改，纠正结果写入 Markdown 笔记的脚注
3. **CorrectionDetector**：每 5 句触发一次批量检查，发现的纠正统一写入笔记脚注

## 原因

1. **字幕回溯改已确认句是视觉灾难**：用户在阅读字幕时，如果已读过的内容突然变化，会造成严重的视觉干扰和认知负担。实验表明，字幕回溯修改会导致：
   - 阅读节奏被打断——用户需要重新扫描已读内容
   - 信任感下降——用户不确定当前看到的是否是最终版本
   - 视觉闪烁——频繁的 DOM 更新导致字幕区域闪烁

2. **笔记脚注既满足比赛"修正能力"要求又不影响体验**：七牛云 XEngineer AI Coding 比赛要求系统具备"修正能力"，即能够纠正翻译错误。笔记脚注方案：
   - 保留了完整的纠正记录，满足比赛评审要求
   - 不干扰字幕阅读体验，用户可以专注于当前内容
   - 事后回顾笔记时，脚注提供了纠正的上下文和对比

3. **"冻结"概念提供清晰的分界点**：管道架构中"句子确认（冻结）"是一个天然的状态边界。冻结前，字幕内容仍在变化，用户预期会看到更新；冻结后，字幕内容固定，用户预期不再变化。纠正策略与这个心理模型完全一致。

4. **CorrectionDetector 批量检查减少干扰**：每 5 句触发一次批量纠正检查，而非每句触发，原因是：
   - 减少纠正检测的频率，降低 LLM 调用成本
   - 批量检查可以利用更多上下文（5 句的完整语义），提高纠正质量
   - 笔记脚注的批量写入比逐条写入更高效

5. **72 小时开发周期下的务实选择**：字幕实时回溯需要复杂的 DOM diff 和动画逻辑，开发量大且容易出 bug。笔记脚注只需在 Markdown 生成时追加脚注内容，实现简单可靠。

## 后果

**正面：**
- 字幕阅读体验稳定流畅，无视觉闪烁和阅读中断
- 笔记脚注保留完整纠正记录，满足比赛评审要求
- 实现简单，开发量小，72 小时内可高质量交付
- 用户心理模型一致：冻结 = 内容固定

**负面：**
- 用户在字幕中无法直接看到已冻结句子的纠正，需要查看笔记
- 笔记脚注可能积累较多，影响笔记的简洁性
- 每 5 句批量检查的粒度可能不够精细，某些明显错误需要等待更长时间才被纠正

**缓解措施：**
- 在字幕区域添加轻微的视觉提示（如小图标），告知用户笔记中有纠正脚注
- 笔记脚注格式统一为 `[^n]` 标记 + 底部对照表，保持笔记主体内容的可读性
- 预留配置项，未来可调整批量检查的触发间隔（如从 5 句改为 3 句）
