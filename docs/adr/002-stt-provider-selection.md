# ADR-002: 选择讯飞作为 STT 服务商

## 状态

已接受

## 背景

SynchroLens 需要实时语音转文字（STT）能力，核心需求为：

- **实时流式输出**：逐词/逐句输出识别结果，而非等语音结束后才返回
- **低首帧延迟**：STT 首字延迟需 < 500ms（PRD 要求）
- **中英文混合识别**：同声传译场景中讲者经常中英混用
- **国内网络可用性**：比赛环境在国内，需确保网络稳定可达

我们评估了三个候选方案：

| 方案 | 传输协议 | 流式输出 | 首帧延迟 | 中英混合 | 国内可用性 | 价格 |
|------|----------|----------|----------|----------|------------|------|
| **讯飞 WebSocket 实时转写** | WebSocket | 逐词 | < 80ms | 优秀 | 原生支持 | 免费额度充足 |
| **阿里云智能语音交互** | WebSocket | 逐句 | ~200ms | 良好 | 原生支持 | 按量计费 |
| **本地 Whisper (whisper.cpp)** | 本地推理 | 整句 | > 1s | 一般 | 无需网络 | 免费（需 GPU） |

## 决策

采用**讯飞 WebSocket 实时转写**作为 STT 服务。

## 原因

1. **首帧延迟最低**：讯飞实时转写首帧延迟 < 80ms，远低于 PRD 要求的 500ms，也为后续翻译环节留出了充足的延迟预算。阿里云约 200ms，本地 Whisper 整句输出延迟 > 1s，均不满足低延迟要求。

2. **逐词流式输出**：讯飞实时转写以"逐词"粒度输出中间结果，翻译引擎可以尽早开始工作，实现"边听边翻"的效果。阿里云为逐句输出，需等待整句识别完成才能开始翻译，增加了端到端延迟。本地 Whisper 只支持整句输出，无法实现流式翻译。

3. **中英文混合识别能力最强**：讯飞在中文语音识别领域深耕多年，对中英文混合场景（如"这个 API 的 response time 很快"）的识别准确率在业内领先。阿里云表现良好但略逊，本地 Whisper 在中文场景下的混合识别效果一般。

4. **WebSocket 协议天然适配管道架构**：WebSocket 长连接与管道式架构的流式数据处理模型完美匹配——音频数据持续推入，识别结果持续流出，无需轮询或额外的事件机制。

5. **国内网络稳定性**：讯飞服务器部署在国内，WebSocket 连接稳定，延迟低。本地 Whisper 虽无网络依赖，但需要 GPU 资源且延迟过高；海外服务（如 Google Speech-to-Text）在国内网络环境下不稳定。

6. **72 小时开发效率**：讯飞提供完善的 WebSocket 实时转写 SDK 和文档，接入代码量少，调试工具齐全。阿里云 SDK 同样成熟，但逐句输出模式需要额外适配。

## 后果

**正面：**
- STT 首帧延迟 < 80ms，为翻译和渲染留出 > 2.9s 的延迟预算
- 逐词输出使翻译引擎可以提前介入，进一步降低端到端延迟
- 中英文混合识别准确，减少后处理纠正负担
- WebSocket 协议与管道架构无缝集成

**负面：**
- 依赖讯飞服务可用性，若服务宕机则 STT 完全不可用
- 讯飞 API 有并发连接数限制，多路音频需额外管理
- 识别结果可能包含"中间纠正"（先输出"你好世"再纠正为"你好世界"），需在管道中处理

**缓解措施：**
- 实现断线重连机制，WebSocket 异常时自动恢复
- 在管道的 STT 阶段处理中间纠正：用讯飞返回的 `pgs` 字段（replace 指令）更新原文流
- 监控 API 用量，避免超出免费额度
